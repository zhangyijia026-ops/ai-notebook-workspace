# AI 笔记本 MVP 工作任务拆解

## 1. MVP 开发目标

第一版目标不是做完整的印象笔记，而是先跑通一个核心闭环：

> 记录任务和笔记 -> 保存灵感摘录 -> AI 整理内容 -> 生成日报/周报。

只要这个闭环能被日常使用，就可以进入下一轮迭代。

## 2. MVP 范围边界

### 本期要做

- 本地优先 Web 应用。
- 笔记本和笔记管理。
- 简单 Todo 管理。
- 笔记内 Todo 识别。
- 摘录收集箱。
- AI 总结、分类、提取 Todo。
- 日报和周报生成。
- 基础搜索。
- 数据本地持久化。

### 本期不做

- 手机 App。
- 浏览器插件。
- 微信和飞书自动同步。
- OCR 截图识别。
- 多人协作。
- 账号体系。
- 付费和订阅。
- 模板市场。

## 3. 推荐技术方案

### 前端

- React + Vite，适合快速搭建本地 MVP。
- TypeScript，便于后续维护数据结构。
- CSS Modules 或普通 CSS，先避免引入过重 UI 框架。

### 本地存储

- MVP 第一阶段优先使用 IndexedDB 或 localStorage。
- 若需要更稳定的数据层，可在第二阶段切换 SQLite/Tauri 或 Supabase。

### AI 接入

- 第一阶段先抽象 `aiService`，让页面逻辑不直接依赖具体模型供应商。
- 可以先使用 mock AI 结果，确保产品流程跑通。
- 接入真实模型时，再添加 API key 配置和后端代理，避免前端泄露密钥。

## Milestone 10：真实 AI API 接入基础设施

目标：在不破坏本地 mock 使用体验的前提下，建立真实 AI API 的接入通道。

任务：

- 增加 AI 模式配置：
  - `mock`：默认模式，无需 API Key。
  - `real`：通过本地代理调用真实 AI。
- 增加 `.env.example`，说明前端和代理需要的环境变量。
- 增加本地 AI 代理服务，避免前端暴露 API Key。
- 支持 OpenAI-compatible 网关配置，例如 One API：
  - `AI_API_KEY`
  - `AI_BASE_URL`
  - `AI_MODEL`
- 前端增加 `aiClient`，统一调用本地代理。
- 将现有 `aiService` 改为 mock/real 可切换。
- 真实 AI 调用失败时自动回退 mock，避免功能不可用。
- AI 输出使用结构化 JSON：
  - 摘录/笔记整理：summary、tags、recommendedNotebookId、extractedTodos。
  - 日报/周报：title、content。

验收标准：

- 不配置 API Key 时，应用仍能使用 mock AI。
- 配置 `VITE_AI_MODE=real`、`AI_API_KEY`、`AI_BASE_URL`、`AI_MODEL` 后，可以通过本地代理调用真实 AI 或 One API。
- `npm run build` 和 `npm run lint` 通过。
- 本地代理可以通过 `npm run dev:ai` 启动。

## 5. 推荐开发顺序

建议按以下顺序推进：

1. Milestone 0：项目初始化。
2. Milestone 2：笔记本和笔记编辑。
3. Milestone 3：Todo 管理。
4. Milestone 4：笔记内 Todo 识别。
5. Milestone 5：摘录收集箱。
6. Milestone 1：首页工作台。
7. Milestone 6：AI 能力第一版。
8. Milestone 7：日报和周报生成。
9. Milestone 8：基础搜索。
10. Milestone 9：UI/UX 整体重构。
11. Milestone 10：真实 AI API 接入基础设施。
12. Milestone 11：真实 AI 效果调优与可控采纳。
13. Milestone 12：笔记命名、笔记本自定义与数据可见性。
14. Milestone 13：Supabase 云同步基础。
15. Milestone 14：项目维度分类与按项目生成报告。

原因：

- 笔记、Todo、摘录是底层数据能力。
- 首页依赖这些数据展示。
- AI 和报告生成依赖已有内容。
- 搜索适合在数据结构稳定后补上。
- UI/UX 重构适合在主要页面和状态都真实存在后统一处理，避免反复返工。
